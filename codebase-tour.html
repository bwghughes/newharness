A compact Rust agent harness

A guided walk through the harness.

This repository is a small terminal coding agent. It speaks the OpenAI-compatible chat API, streams model output as it arrives, gives the model a narrow set of filesystem and shell tools, and records the plan in both a terminal/web board and TASKS.md.

Runtime Flow

The program keeps the loop deliberately short: read a request, stream a model turn, run any requested tools, append tool results, and continue until the assistant stops asking for tools.

CLI main.rs reads env config, prints the prompt, and accepts slash commands.
Agent agent.rs owns message history, plan parsing, turn limits, and tool dispatch.
Client client.rs posts chat completions, streams SSE, and assembles tool-call deltas.
Tools tools.rs exposes read, list, edit, grep, bash, and Tavily web search.
Board plan.rs and web.rs broadcast progress and append history.

Design Decisions

The harness favors small, explicit pieces over a framework-heavy architecture. Most choices are there to keep agent behavior observable and provider integration predictable.

Implementation Walkthrough

This is the path a normal prompt takes through the system, from terminal input to model stream to filesystem changes.

Configuration is selected up front.

read_config chooses the base URL, model, workdir, verbosity, and API key. For the ai-stack LiteLLM endpoint it prefers LITELLM_VIRTUAL_KEY, then LITELLM_MASTER_KEY, then generic keys, and rejects placeholder values like sk-change-me.

The system prompt is built from code plus local policy.

Agent::new loads tool schemas, installs the built-in coding-agent instructions, and appends the nearest parent STRAP.md when one exists.

The client streams and assembles partial responses.

StreamAssembler accumulates text, sparse tool-call indexes, function names, and argument fragments until [DONE] arrives.

Tool calls run through a narrow executor.

Single calls get a spinner; multiple calls are spawned concurrently with Tokio and returned to the model as tool messages.

Plan state is broadcast and archived.

Numbered assistant plans become board tasks. The web UI receives SSE updates, and TASKS.md gets a timestamped completion record at the end of the turn.

The agent loop has a hard cap of 50 model/tool turns. That is a practical guardrail: enough room for real work, but finite when a model keeps asking for tools.

Testing Notes

The tests are mostly unit tests around the contract edges: serialization, streaming assembly, tool behavior, config normalization, and web board event shape.

Area What the tests protect Why it matters
client.rs SSE parsing, token usage, tool-call deltas, and auth header formatting. Provider quirks show up here first, especially with LiteLLM and streamed tool calls.
tools.rs Exact file edits, path handling, grep behavior, fallback search, and command execution output. The model only changes the world through these tools, so small edge cases can become big agent failures.
plan.rs Numbered-plan parsing, task progression, usage accounting, and TASKS.md output. The board is the user's window into what the agent thinks it is doing.
web.rs Embedded board HTML, SSE event construction, initial snapshots, and lag recovery. The browser view should stay useful even when subscribers connect late or miss events.
